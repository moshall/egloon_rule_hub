# Rule Directory Publishing Design

## Goal

Change the publish surface of `egloon_rule_hub` from flat per-target files and service detail pages into target-scoped rule directories that mirror the browsing style of upstream rule repositories.

After this change, the main user-facing entrypoint for a rule should be the directory itself, for example:

- `Rule/Clash/OpenAI/OpenAI.yaml`
- `Rule/Clash/OpenAI/README.md`
- `Rule/Egern/OpenAI/OpenAI.yaml`
- `Rule/Egern/OpenAI/README.md`

The directory README must stay fact-based:

- native target outputs should show the matching upstream README when one exists
- converted target outputs should explicitly say they are generated by this repository
- upstream README text should still be shown as original source text, not rewritten into a fake unified upstream document

## Scope

This slice changes the published layout for per-service artifacts and their documentation.

It should cover:

- replacing flat per-target service artifact paths with `Rule/<Target>/<Service>/`
- generating one `README.md` per target/service directory
- rendering target-specific provenance instead of service-level aggregate pages
- keeping upstream README snapshots as internal build artifacts
- updating tests and CLI output expectations to the new publish surface

It should not cover:

- redesigning the service catalog model
- changing the multi-source merge model
- redesigning bundle outputs in this same slice
- backward-compatible migration paths for old file locations

## User-Facing Outcome

After `python -m egloon_rule_hub bootstrap`, the repository should publish per-service rule outputs in directory form.

Examples:

- `Rule/Clash/OpenAI/OpenAI.yaml`
- `Rule/Clash/OpenAI/README.md`
- `Rule/Loon/OpenAI/OpenAI.list`
- `Rule/Loon/OpenAI/README.md`
- `Rule/Egern/OpenAI/OpenAI.yaml`
- `Rule/Egern/OpenAI/README.md`

The existing flat `dist/<target>/<Service>.<ext>` layout should no longer be treated as the primary per-service output surface.

Internal machine-readable artifacts may continue to live under `dist/`, especially:

- `dist/manifests/*.json`
- `dist/upstream-readmes/`
- any unchanged bundle outputs during this slice

## Core Decisions

### 1. Directory publishing replaces flat service files

Per-service publish paths should move from flat target buckets such as:

- `dist/clash/OpenAI.yaml`
- `dist/egern/OpenAI.yaml`

to target-scoped directories:

- `Rule/Clash/OpenAI/OpenAI.yaml`
- `Rule/Egern/OpenAI/OpenAI.yaml`

This should be a direct rebuild of the output surface, not a compatibility migration, because the repository has not yet been formally integrated as a stable downstream dependency.

### 2. The directory README is the primary documentation view

Each `Rule/<Target>/<Service>/README.md` should describe exactly that target-specific artifact.

The repository should no longer treat `docs/services/<Service>.md` as the primary publish surface for rule usage documentation.

This makes the directory itself self-contained for browsing, linking, and human inspection.

### 3. README rendering is target-aware, not just service-aware

The documentation for `Clash/OpenAI` and `Egern/OpenAI` should not be assumed to be identical.

Each target-specific README should be rendered from the perspective of the actual artifact being published in that directory:

- native upstream target when available
- converted target when generated locally from one or more other upstreams

### 4. Converted targets must not pretend to be native upstream outputs

Targets such as `Egern` may be generated from `Clash`, `Loon`, or other upstream-native inputs.

In those cases, the directory README must explicitly say that:

- the rule file in this directory is generated by `egloon_rule_hub`
- the file is not a native upstream target artifact
- the actual upstream sources used for generation are listed below

This keeps the directory structure convenient without blurring authorship or provenance.

### 5. Multi-source services stay source-separated

If a target/service artifact is built from multiple upstream sources, the README may contain one short local summary at the top, but the upstream original text must remain grouped by source block.

The system must not synthesize a fake canonical upstream README by merging source prose together.

### 6. Missing upstream README stays explicit

If an upstream rule input exists but the derived `README.md` does not, the target README should say `upstream README missing`.

The renderer must not infer missing text from repository homepages, neighboring folders, or naming conventions.

## Output Structure

### Rule directories

The primary publish layout for per-service rules should be:

`Rule/<Target>/<Service>/<Service>.<ext>`

and:

`Rule/<Target>/<Service>/README.md`

Target names should use fixed display casing:

- `Clash`
- `Egern`
- `Loon`
- `QuanX`
- `Shadowrocket`

Service directory names should continue to use the catalog service name as-is, for example `OpenAI`, `GoogleVoice`, `ChinaIPsBGP`.

### Rule filenames

The rule filename inside the directory should default to the service name plus the target extension:

- `OpenAI.yaml`
- `OpenAI.list`

### Internal snapshots

Upstream README snapshots should still be written as internal build artifacts under:

`dist/upstream-readmes/<service>/<source>/README.md`

The snapshot layer remains an implementation detail used to render the published directory READMEs deterministically.

### Internal manifests

Manifest output may continue to live under:

`dist/manifests/`

This slice does not require a new public manifest file inside each rule directory.

## README Content Contract

Each `Rule/<Target>/<Service>/README.md` should render in this order:

1. target-specific title, for example `# OpenAI for Clash`
2. file information
3. source classification
4. upstream references
5. local summary
6. upstream original text blocks
7. factual notes for missing or fetch-error states

### File information

This section should state at minimum:

- service name
- target name
- rule filename

### Source classification

If the target is backed by a native upstream artifact for that target, the README should say so plainly.

If the target is generated by conversion, the README should say so plainly, for example:

`This rule file is generated by egloon_rule_hub and is not a native upstream Egern artifact.`

### Upstream references

This section should list:

- the exact upstream rule file URLs used to build this target/service artifact
- the exact upstream README URLs derived from those inputs
- source ordering when multiple upstreams are involved

### Local summary

This section may briefly describe how the current target artifact was produced, for example:

- generated from Clash and Loon upstream rule sets
- normalized and deduplicated before target emission
- merged by configured source priority

The summary should be local and factual, not a rewrite of upstream README prose.

### Upstream original text

Original upstream README text must be shown as-is in source-separated blocks.

Examples:

- one native Clash upstream block for `Rule/Clash/OpenAI/README.md`
- multiple source blocks for `Rule/Egern/OpenAI/README.md` when Egern output is derived from multiple upstream-native sources

If the upstream README is missing, render `upstream README missing`.

If fetch failed transiently, render `upstream README fetch_error`.

## Data Flow

The rule and README generation flow should be:

1. load catalog definitions
2. resolve upstream rule sources as usual
3. build normalized service rule streams
4. render target-specific rule files into `Rule/<Target>/<Service>/`
5. fetch and snapshot upstream README content for the actual resolved upstream inputs
6. render target-specific directory READMEs from snapshot data and resolved source metadata
7. render machine-readable manifests under `dist/manifests/`

The important change is that README rendering should now be keyed by both:

- target
- service

rather than only by service.

## Target-Specific README Strategy

### Native target case

If a target/service artifact is backed by a native upstream file for the same target, the README for that directory should use that target's upstream README as the primary upstream text source.

Example:

- `Rule/Clash/OpenAI/OpenAI.yaml`
- upstream rule: `rule/Clash/OpenAI/OpenAI.yaml`
- upstream README: `rule/Clash/OpenAI/README.md`

### Converted target case

If a target/service artifact is generated from upstream inputs that are native to other targets, the README for that directory should:

- say that the artifact is generated locally
- enumerate the actual upstream rule files used
- enumerate the actual upstream README URLs used
- include original upstream README text grouped by source block

Example:

- `Rule/Egern/OpenAI/OpenAI.yaml`
- generated from one or more upstream-native sources such as Clash or Loon
- README explains the conversion relationship instead of pretending there is a native upstream Egern README

### Multi-source case

When multiple upstream inputs participate in building the target artifact:

- preserve priority ordering in the references section
- show separate source blocks for original README text
- keep the local summary short and factual

## File Boundaries

### `src/egloon_rule_hub/build.py`

Update rule artifact output paths so per-service files are written into:

- `Rule/<Target>/<Service>/<Service>.<ext>`

This module remains responsible for rule emission, not README prose rendering.

### `src/egloon_rule_hub/upstream_docs/build.py`

Extend or adapt the upstream docs manifest so README tracking can be queried by target/service output needs, not only service-level aggregate needs.

This module remains the boundary between upstream README fetches and deterministic local snapshot output.

### `src/egloon_rule_hub/docs/render.py`

Change docs rendering responsibility from service detail pages under `docs/services/` to target-specific directory READMEs under:

- `Rule/<Target>/<Service>/README.md`

This module should render markdown from local manifest and snapshot data only.

It should not make network requests.

### `src/egloon_rule_hub/cli.py`

`bootstrap` should continue to orchestrate:

- catalog validation
- rule rendering
- manifest rendering
- upstream README snapshot generation
- target-specific README rendering

The CLI surface does not need a new top-level command for this slice.

## Error Handling

### Rule fetch failure

If an upstream rule file required for artifact generation cannot be fetched, bootstrap should still fail. That is a core pipeline failure.

### README missing

If the derived upstream `README.md` does not exist, record `missing` and continue.

The published target README should render that state explicitly.

### README transient fetch error

If README fetching fails due to timeout, rate limiting, or other temporary upstream issues, record `fetch_error` and continue.

The published target README should distinguish this from `missing`.

## Testing Strategy

### Rule output tests

Update tests so they verify directory outputs such as:

- `Rule/Clash/OpenAI/OpenAI.yaml`
- `Rule/Egern/OpenAI/OpenAI.yaml`
- `Rule/Loon/OpenAI/OpenAI.list`

Flat per-target output assertions should no longer be the main expectation for per-service files.

### README rendering tests

Add tests that confirm:

- a native target README is written to `Rule/<Target>/<Service>/README.md`
- a converted target README is written to `Rule/<Target>/<Service>/README.md`
- converted target README text clearly says the artifact is generated locally
- multi-source README rendering keeps upstream original text split by source block
- missing README state is rendered explicitly

### Integration-style checks

After `bootstrap`, verify representative paths such as:

- `Rule/Clash/OpenAI/OpenAI.yaml`
- `Rule/Clash/OpenAI/README.md`
- `Rule/Egern/OpenAI/OpenAI.yaml`
- `Rule/Egern/OpenAI/README.md`

Fixture-based tests should continue to avoid live-network dependence in CI.

## Workflow Integration

The existing GitHub Actions workflows should keep running `bootstrap`.

The bot commit behavior may now produce diffs under:

- `Rule/`
- `dist/`
- `docs/` when non-directory docs still exist

This slice intentionally allows a direct rebuild of the generated publish surface rather than a staged compatibility migration.

## Acceptance Criteria

This design is complete when:

- per-service rule outputs are published under `Rule/<Target>/<Service>/`
- each target/service directory contains both the rule file and `README.md`
- native target READMEs reflect the matching upstream target README when available
- converted target READMEs clearly state they are generated artifacts
- converted target READMEs list actual upstream rule and README references
- upstream original text remains source-separated
- missing upstream README is shown explicitly
- `bootstrap` regenerates the new structure deterministically in CI
